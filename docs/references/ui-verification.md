# UI Verification Strategy (Browser Automation)

フロントエンドコンポーネントの大幅な変更や新規追加を行った場合、単にリンターやビルドコマンドを通すだけでなく、「Agentic UI Verification」によって視覚的かつ動的なテストを実施する「自己改善ループ」を回します。

## 1. 検証用サーバーの立ち上げ
UIの変更が完了した後、まだ立ち上がっていない場合はターミナルで `bun run dev` (または `nix develop -c bun run dev`) を実行して、ローカル開発サーバーを起動します。

## 2. Browser Subagent の起動とタスク付与
エージェント自身のツールである `browser_subagent` を起動し、以下の観点での視覚的チェックタスクを依頼します。

### 検証観点 (Verification Points)
1. **Design System & Contrast (`DESIGN.md` の遵守)**
   - 変更されたページにアクセスし、文字と背景のコントラスト比が確保されているか確認する。
   - `px` ベースのマジックナンバーによる不自然な余白のずれがないか、Tailwind のトークンがきれいに並んでいるかを確認する。
2. **Responsive Checks**
   - ウィンドウサイズをモバイルサイズ等に変更させ、レイアウトが画面領域を突き破っていないか（Overflowしていないか）を確認する。
3. **Interactive Elements**
   - 作成したボタン、アコーディオン、ダイアログ等をクリックさせ、アニメーションや状態変化が意図した通りか、意図せぬエラーやコンソール出力が出ないか確認する。

## 3. フィードバックループの適用
ブラウザサブエージェントから「テキストが背景に溶け込んで見えない」「クリックしてもドロップダウンが開かない」等のフィードバックが返ってきた場合、それを「Harness Engineeringへの違反」とみなし、即座に修正のイテレーションを回します。

## 4. 人間への最終報告
問題が解決した段階で `walkthrough.md` に視覚的検証を行った動画・スクリーンショットを埋め込み、必要に応じて GitHub PR Workflow を通じて変更を提出します。
