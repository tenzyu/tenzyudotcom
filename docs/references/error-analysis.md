# Error Analysis Skill & Workflow

エージェントがビルドエラーやランタイムエラーに直面した際は、場当たり的な「とりあえず動く修正（YOLO）」を行うのではなく、以下の「エラー分析ループ（自己改善ループ）」を回してナレッジを蓄積します。

## 1. エラーの根本原因の特定 (Root Cause Analysis)
エラーに直面した際、すぐにコードを書き換えるのではなく、まずはログとトレースを深く分析します。
- `vercel logs` や 開発サーバー（`bun run dev`）からのスタックトレースを読み込む。
- どのコンポーネントや関数がエラーの起点となったかを特定する。
- それが「データ構造のエラー（Zod等）」なのか「クライアント/サーバー境界の違反（RSC Payload等）」なのか分類する。

## 2. ゴールデンプリンシプル（規律）との照合
エラーの原因が判明した場合、既存のHarness Engineeringの規律（`docs/` 配下のポリシー）と照らし合わせます。
- **`RELIABILITY.md` (障害耐性)**: エラーがアプリケーション全体をクラッシュさせた場合、`ErrorBoundary` がカプセル化（Isolated Error Boundaries）されていなかったか確認し、Fail-Closed に倒すべきか Fail-Open (not-found) にするか判断する。
- **`QUALITY_SCORE.md` (型と境界)**: API境界でのバリデーション漏れによる TypeError の場合は、Zodスキーマへの差し込みを行う。

## 3. 修正のテストと検証
修正方針が決まった段階でコードを書き換え、`bun run verify` 等を実行して問題が完全に解決されたか確認します。必要であればブラウザサブエージェントを使って視覚的に確認（例: ErrorBoundary のフォールバック画面の表示）します。

## 4. ナレッジ化（Case Study の作成）
最も重要なステップとして、修正が完了した後に同じ過ちをエージェントが繰り返さないよう、今回のエラーと解決の軌跡を「ケーススタディ」として永続化させます。

- `docs/exec-plans/completed/` ディレクトリに新しいMarkdownファイル（例: `fix-hydration-error.md`）を作成する。
- 発生した問題、取られた戦略、そして参考になった公式ドキュメント（Sources）を記載する。
- これにより、プロジェクトの自己改善ループ（Small Start Harness）が成立します。
