# Reliability (RELIABILITY)

このドキュメントは、システム全体の継続稼働を担保するためのエラーと障害耐性の**ゴールデンプリンシプル**である。エージェントは機能を作成する際、それらが失敗した時のフォールバックを想定すること。

## 1. Fault Tolerance Boundaries (障害耐性と境界の局所化)

サイトの一部（単一の機能やツール）でのエラーが、アプリケーション全体や別ルートをクラッシュさせることを防ぐ。

- **Isolated Error Boundaries**:
  - `tools-product-spec.md` に見られるように、独立したアプリケーションや機能モジュール（例: `/tools/*` 内のミニアプリ）は、個別に稼働する機能単位であるため、必ず自立したエラー境界（`ErrorBoundary`）を持つこと。
  - レンダリング時の例外やクライアント状態の崩壊が、システム全体のページ表示やグローバルレイアウト（ヘッダー・フーターの表示など）まで巻き込んでクラッシュさせないようにすること。

## 2. Fallback Strategies (フォールバック戦略)

特定のエラー条件（Not Found, Invalid Data, Timeout, Storage API outage）に基づくデータ不在への耐性に関するポリシー。

-ストレージやCMSからの取得失敗時は、`not-found` を返すのか、一時的な `outage/invalid-data` としてフォールバックコンポーネントを表示させるのかを明確に定義し、常に最悪のケースに備えてハンドリングする。
- **Fail-Openの制限（Not Foundのみ許容）**: データが見つからなかった場合（Not Found）は、UI側で空状態やデフォルトのフォールバック画面を表示するFail-Openを許容する。
- **Fail-Closedの強制（無効データ・障害時）**: スキーマ検証エラーやバックエンドAPI（BlobやDB）の完全なダウン時においては、データの損傷や誤動作（例えば保存による上書き）を防ぐため、安全に Reject し、ユーザー（またはAdmin）に再試行を促す「Fail-Closed」として扱う。不要な自動リトライと無効データの保存を避ける。

## 3. レンダリングとプロキシの安全性

- **Client vs Server**: 必要のないステートフックやブラウザAPIへの依存をClient Componentsへ分離する。サーバーサイドのHTMLとしてプレレンダリング可能な部分は最大限活用し、初期ロードの信頼性を高める。
- **Front-door Validation**: フレームワークのエントリーとしての `proxy.ts` は、ロケールの確定・ルートの書き換え（Rewrite/Redirect）等の初期リクエスト検証にのみ限定し、機能のドメインロジックを持たせないことで、ルーティング自体の信頼性とパフォーマンスを確保する。

## 4. Next.js App Router Reliability (メタデータとi18nの確実性)

Next.js App Router において、ルーティングやSEOに深く関わるファイル郡の確実性を担保する。

- **Metadata Safety (メタデータの安全性)**:
  - `sitemap.ts` や `robots.ts` のような重要なメタデータを生成する際、不確かなロケール推論やリクエストのヘッダー情報（Hostなど）による相対的解決に頼らないこと（`nextjs-metadata-i18n.md` の教訓に基づく）。
  - ベースとなるサイトURL（Base URL）は、常にビルド時または実行時に確定された単一の環境変数（`SITE_URL` 等）から明示的に注入し、正しい絶対パスを出力すること。
  - 不確実性のあるミドルウェアや非推奨なAPIを「Precedent」として真似ないこと。
- **i18n Metadata**:
  - すべてのサポートロケールリストを明示的にループし、構築フェーズ（ビルド時・リクエスト時）で正確なURLや `alternates` がすべての言語間で漏れなく生成されることを保証する。
